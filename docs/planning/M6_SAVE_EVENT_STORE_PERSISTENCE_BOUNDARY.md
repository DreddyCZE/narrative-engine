# M6 Save / Event Store / Persistence Boundary

## 1. M6 Goal

M6 exists to design a safe Save / Event Store / Persistence boundary above the accepted M5 runtime output.

The goal is to:

- define how committed runtime results and return-only runtime domain event values can be persisted later
- preserve the runtime host as a pure and in-memory boundary
- keep Save, Event Store, persistence, replay, and storage concerns outside runtime execution
- avoid UI/editor, gameplay/P0 content, plugin runtime, and production file-loader scope

M6 planning does not implement persistence behavior. It defines the boundaries, envelopes, result contracts, and task sequence required for future implementation.

## 2. Why M6 Exists

M4 delivered a validated content graph boundary over caller-provided content package objects.

M5 delivered:

- runtime host input/result contracts
- runtime command request resolver
- runtime condition/effect binding adapter
- in-memory command execution pipeline
- return-only runtime domain event values
- minimal fixture runtime command integration coverage

That gives the repository a deterministic, pure, in-memory runtime execution boundary. M6 exists to design how those outputs can be stored and later restored without violating the runtime boundary, mixing persistence into the runtime host, or introducing premature file or database IO.

## 3. Architecture Boundary

### Runtime host

The runtime host remains responsible for:

- consuming `RuntimeHostInput`
- returning `RuntimeHostResult`
- returning explicit `runtimeDomainEventValues`
- keeping execution deterministic and in memory
- avoiding storage writes

The runtime host must not:

- write an Event Store
- write Save snapshots
- choose storage backend details
- perform file IO or database IO
- own replay orchestration

### Persistence boundary

The persistence boundary is a future layer above runtime execution.

It should:

- receive explicit persistence requests or committed runtime outputs
- validate persistence envelopes before write attempts
- serialize committed state snapshots and event records deterministically
- return explicit persistence results and diagnostics
- remain separate from UI/editor and gameplay flows

### Storage adapter boundary

The storage adapter is a future optional layer below persistence orchestration.

It should:

- expose backend-specific append, save, load, and listing capabilities
- remain isolated from runtime host types and logic
- support in-memory first, then file/JSON, then database options in later tasks

No concrete adapter is introduced in this planning task.

### UI / editor / plugin boundary

UI, editor, and plugin surfaces remain outside M6 planning scope except as future consumers of the persistence boundary. They must call explicit persistence interfaces rather than reach into runtime host internals.

## 4. Event Store Boundary

M6 should define an append-only Event Store boundary that consumes explicit event records derived from committed runtime outputs.

### Core rules

- Event Store append is a persistence concern, not a runtime host concern.
- Only committed runtime results may produce appendable event records.
- Rejected, blocked, error, and no-op runtime outcomes must not append confirmed event records.
- Event records must remain deterministic and JSON-safe.

### Proposed event envelope

Each future event record should include:

- `eventId`
- `eventType`
- `sourceCommandId`
- `transactionId` when available
- `runtimeMetadata`
- `payload`
- `schemaVersion`
- `createdAtPolicy` or deterministic timestamp policy
- `persistenceMetadata`

`createdAt` should not be silently generated by runtime execution. The plan should allow either:

- an explicit deterministic logical timestamp policy for test and replay-safe flows, or
- a storage-layer metadata field outside canonical event meaning

### Validation before append

The persistence boundary should validate:

- event envelope shape
- canonical event identity rules
- allowed metadata shape
- payload JSON safety
- schema version support
- duplication and idempotency expectations

### Idempotency and deduplication

The Event Store plan should include:

- append idempotency keyed by `eventId` or explicit idempotency key
- duplicate append detection with deterministic `blocked` or `rejected` outcomes
- explicit diagnostics for conflicting duplicate content

No direct runtime writes are allowed. Runtime only returns values; persistence chooses whether and how to append them.

## 5. Save Snapshot Boundary

M6 should define a save snapshot boundary that stores committed Engine State snapshots in an explicit Save envelope.

### Proposed snapshot envelope

Each save snapshot should include:

- `snapshotId`
- `stateId`
- `revision`
- `sourceEventRange`
- `contentPackageId`
- content package or manifest version references
- runtime version reference
- schema or contract version references
- checksum or hash metadata
- persistence metadata
- canonical Engine State snapshot payload

### Save boundary rules

- Save snapshots are derived from committed Engine State only.
- Save snapshot persistence must validate runtime/content compatibility before write.
- Save snapshot persistence must not mutate runtime state.
- Save UI flow remains out of scope.
- Save snapshot writes must remain outside the runtime host.

### Versioning and integrity

The Save boundary plan should account for:

- content package identity and version references
- runtime version references
- checksum or hash strategy for snapshot integrity
- future migration policy when save format or contract versions change

## 6. Persistence Result Contract

M6 should define future persistence result statuses with deterministic metadata and diagnostics:

- `saved`
- `appended`
- `loaded`
- `rejected`
- `blocked`
- `error`

### Result behavior

- `saved`: a snapshot envelope was validated and stored successfully
- `appended`: event record or event batch append succeeded
- `loaded`: a snapshot or event-set load succeeded
- `rejected`: the request was structurally valid but refused by version, integrity, or compatibility rules
- `blocked`: the request could not proceed because prerequisites were missing or ambiguous
- `error`: an unexpected validation or adapter failure occurred

### Result metadata

The future contract should include:

- deterministic result metadata
- stable diagnostic ordering
- explicit source and boundary labels
- storage-neutral summaries
- optional backend metadata kept outside canonical runtime meaning

## 7. Replay Strategy

Replay remains planning-only in M6.

The plan should support two future replay entry points:

- rebuild state from event records only
- rebuild state from a saved snapshot plus subsequent event records

### Replay expectations

- replay inputs must be explicit and version-aware
- replay must preserve determinism expectations established by M5
- replay diagnostics must clearly separate invalid persistence data from runtime rule failures
- replay must not bypass the current Engine State contract

Replay implementation remains deferred until later M6 tasks or a later milestone if the task sequence needs further decomposition.

## 8. Storage Adapter Strategy

The storage adapter strategy should be incremental.

### First adapter

- in-memory persistence adapter first
- deterministic test-only behavior
- no production file IO

### Later adapters

- JSON or file adapter later in an explicit storage task
- database adapter later in a separate future task
- no adapter implementation inside runtime host packages

### Separation rule

The storage adapter interface must remain separate from the runtime host. Production file IO is allowed only in a future explicit storage adapter task, never in runtime execution code.

## 9. Test Strategy

M6 should plan tests for:

- append valid return-only event values to an in-memory Event Store boundary
- reject invalid event envelope shape or invalid event metadata
- save a valid committed Engine State snapshot
- reject mismatched content-package or runtime version references
- verify deterministic IDs and metadata
- verify no runtime input or runtime result mutation
- verify runtime host never writes storage directly
- verify replay smoke behavior only if replay is explicitly scoped later

Tests should continue the current repo style:

- pure/data-only first
- in-memory first
- deterministic diagnostics
- minimal fixture coverage before broader matrices

## 11. Non-Goals

M6 planning explicitly excludes:

- production database implementation
- production file IO unless a future explicit storage adapter task approves it
- UI save/load flow
- gameplay/P0 content
- plugin runtime
- runtime host storage writes
- external network storage
- auth or multi-user cloud saves
- Save or Event Store implementation in this planning task

## 12. Risks and Open Questions

Key risks and open questions:

- how to guarantee deterministic behavior when timestamps or persistence metadata appear
- how to bind event records to content package identity and version safely
- how to implement idempotency without leaking persistence concerns into runtime execution
- how to keep runtime result generation separate from persistence write orchestration
- how to prevent Save/Event Store logic from entering the runtime host
- how to handle schemaVersion migrations for snapshots and event records
- how to ensure future replay does not violate the current Engine State contract
- how to represent source event ranges for snapshots when a global Event Store position is not yet defined
